## 2026-03-31 实现 gf doc 全局函数查询并支持 --build-json

### What

1. 实现 `gf doc FUNC`
2. 增加 `gf doc --build-json`
3. 让 `tests/function-library-index.json` 不再依赖外部手工生成脚本
4. 修复若干历史测试文件命名不完全规范时的文档定位问题

当前已经支持的调用形式例如：

```bash
gf doc string-split
gf doc "option=?"
gf doc "bag-search!"
gf doc remove
gf doc "alist->fxmapping/combinator"
gf doc --build-json
```

### Why
阶段 3 完成后，`gf doc` 还缺少真正面向用户的单参数查询能力：

1. 用户通常更自然地输入函数名，而不是先写完整库名
2. 多库实现函数需要依赖索引文件做歧义判断
3. 如果 `function-library-index.json` 仍然依赖仓库外部脚本生成，就会让维护链路断开

因此这次的目标不是只把 `gf doc FUNC` 接上，而是把：

1. 索引读取
2. 索引生成
3. CLI 分发回退
4. 历史命名兼容

一起收口成项目内可自举的一套实现。

### How
本次主要新增了两个单职责模块：

1. `tools/golddoc/liii/golddoc-index.scm`
   - 负责读取 `function-library-index.json`
   - 负责把 `"(liii string)"` 这样的索引项转换成 `"liii/string"`
   - 负责按当前 `*load-path*` 过滤出可见实现库

2. `tools/golddoc/liii/golddoc-index-build.scm`
   - 负责扫描当前 `*load-path*` 可关联到的 `tests` 根目录
   - 负责提取叶子测试文件中的函数文档标题
   - 负责生成并写出 `tests/function-library-index.json`

其中索引写出路径已经改成使用 `(liii njson)`：

1. 先构造普通 Scheme JSON 值
2. 再调用 `json->njson`
3. 最后使用 `njson->file`

也就是说，`golddoc` 自己不再直接依赖 `(liii json)` 去写索引文件。

在 CLI 层，这次做了两个关键补充：

1. `tools/golddoc/liii/golddoc-cli.scm`
   - 新增 `run-function-query`
   - 将 `gf doc FUNC` 接成真实逻辑
   - 新增 `gf doc --build-json`

2. 单参数回退规则
   - 对于 `alist->fxmapping/combinator` 这种“语法上像 `ORG/LIB`，但其实是函数名”的输入
   - 先按库查询尝试
   - 若当前 `*load-path*` 中不存在该库，则自动回退为函数查询

在函数文档定位层，这次还补了两类兼容：

1. 历史 stem 兼容
   - 如 `truncate/ -> truncate-slash-test.scm`

2. 扫描兜底
   - 若标准 stem 拼接失败，则扫描同库目录下的 `*-test.scm`
   - 通过文档标题或语法块反查函数名
   - 这样像 `option=? -> option-equal-p-test.scm`
   - 以及 `bag-search! -> bag-search-test.scm`
   - 这类历史文件也能被 `gf doc` 命中

此外还修了一个底层路径问题：

1. `tools/golddoc/liii/golddoc-library.scm`
   - `find-tests-root-for-load-root` 现在会先去掉尾部路径分隔符
   - 避免 sibling `tests` 路径里出现双斜杠
   - 这能修复 `listdir` 在某些绝对路径上的异常行为

## 2026-03-31 实现 gf doc 精确函数查询

### What

本次新增能力为：

1. 实现 `gf doc ORG/LIB FUNC`
2. 将函数名按照统一规范映射到叶子测试文件名
3. 直接输出对应叶子测试文件内容

当前已经支持的调用形式例如：

```bash

```

### Why
阶段 1 和阶段 2 已经打通了：

1. `gf doc` 宿主入口
2. `gf doc ORG/LIB` 顶层库文档查询

但如果没有阶段 3，`gf doc` 仍然无法命中真正最常用的“单函数文档”场景。也就是说，用户虽然可以看到：

```text

```

这类顶层说明文件，但还不能直接查到：

```text

```

这样的叶子测试文档。

因此阶段 3 的目标就是把“库内某个导出名”准确地落到对应的 `*-test.scm` 文件上。

1. `tools/golddoc/liii/golddoc-function.scm`

这个文件只负责两件事：

1. `exported-name->test-stem`
   - 将导出名转换成测试文件 stem

   - `- -> minus`
   - `* -> star`
   - `/ -> slash`
   - `= -> eq`
   - `< -> lt`
   - `<= -> le`
   - `> -> gt`
   - `>= -> ge`

然后在 `tools/golddoc/liii/golddoc-cli.scm` 中，把原本“未实现”的：

1. `library-function`

分支接成真实逻辑：

1. 解析 `ORG/LIB`
2. 检查是否属于当前排除目录
3. 在当前 `*load-path*` 中定位可见库
4. 推导关联的 `tests` 根目录
5. 将函数名规范化后拼出：
   - `tests/<group>/<library>/<stem>-test.scm`
6. 找到后直接输出该文件全文

同时 `tools/golddoc/liii/golddoc.scm` 也更新为导入并导出新的函数模块能力。

## 2026-03-31 新增 gf doc 子命令脚手架并实现精确库查询

### What
本次改动实现了 `gf doc` 的前两个阶段：

1. 打通 `gf doc` 子命令入口
2. 实现 `gf doc ORG/LIB` 的精确库查询

当前已经支持的调用形式为：

```bash
gf doc liii/string
gf doc scheme/base
```

当前这类调用会：

1. 根据 `*load-path*` 查找当前可见库
2. 推导与该 `load root` 关联的 `tests/` 根目录
3. 定位并输出对应的顶层库说明文件：
   - `tests/<group>/<library>-test.scm`

例如：

```bash
gf doc liii/string
```

当前会输出：

```text
tests/liii/string-test.scm
```

的文件内容。

### Why
这次不是一次性把完整 `gf doc` 做完，而是先把最小可用路径打通，原因有三点：

1. `gf doc` 需要先验证“C++ 子命令分发 + Scheme 工具模块”这条架构在当前工程里是稳定可行的
2. 文档查询的核心边界是 `*load-path*`，因此应先把“当前可见库 -> 顶层文档文件”的定位逻辑做准
3. 后续函数级查询、JSON 索引和模糊匹配都依赖稳定的底层路径推导，如果阶段 1/2 不先站稳，后面会反复返工

### How
本次实现沿用了现有 `gf test` / `gf fix` 的模式：

1. C++ 层只做薄分发
2. Scheme 层负责业务逻辑

具体改动如下：

1. 在 `src/goldfish.hpp` 中：
   - 新增 `find_golddoc_tool_root`
   - 在 `display_help()` 中加入 `doc ORG/LIB`
   - 新增 `command == "doc"` 分支
   - 运行时将 `tools/golddoc` 加入 `*load-path*`
   - 导入 `(liii golddoc)` 并调用 `main`

2. 在 `xmake.lua` 中：
   - 新增 `tools/golddoc/(liii/*.scm)` 的安装规则

3. 在 `tools/golddoc/liii/` 下按职责拆分实现，而不是把全部逻辑堆到单个文件：
   - `golddoc-args.scm`
     - 负责 `gf doc` 参数解析
     - 负责区分 `library` / `library-function` / `function`
   - `golddoc-library.scm`
     - 负责 `*load-path*` 中的可见库查找
     - 负责 `tests root` 推导
     - 负责顶层库文档路径定位
   - `golddoc-cli.scm`
     - 负责 usage、错误输出和 `run-golddoc`
   - `golddoc.scm`
     - 只做聚合导出和 `main`

### 阶段 2 的库查询逻辑
`gf doc ORG/LIB` 当前按以下顺序工作：

1. 解析 `ORG/LIB`
2. 在当前 `*load-path*` 中按顺序查找：
   - `<load-root>/<org>/<lib>.scm`
3. 第一个命中的 `load root` 视为当前真正可见的库
4. 对该 `load root` 推导测试根目录：
   - 先尝试 `<load-root>/tests`
   - 再尝试 `<parent(load-root)>/tests`
5. 定位：
   - `<tests-root>/<group>/<library>-test.scm`
6. 直接输出该文件全文

这使得 `gf doc` 在阶段 2 已经与 `import` 的可见性模型保持一致，而不是简单地扫描整个仓库。

### 范围与边界
本次实现只完成了阶段 1 和阶段 2，因此当前边界是明确的：

1. 已支持：
   - `gf doc ORG/LIB`

2. 还未实现：
   - `gf doc ORG/LIB FUNC`
   - `gf doc FUNC`
   - JSON 索引
   - 模糊匹配

3. 当前排除目录仍保持为：
   - `tests/srfi`
   - `tests/goldfish`

因此目前这些命令会明确报错而不是静默失败：

```bash

gf doc srfi/1
gf doc
```
