## 2026-03-27 新增 `(liii string)` 的 `string-split`

### What
在 `(liii string)` 中新增顶层函数：

```scheme
(string-split str sep)
```

语义定义为：

1. `sep` 支持字符串分隔符，也兼容单个字符
2. 按精确分隔符切分，而不是按 token 化语义压缩分隔符
3. 保留空字段
4. `sep = ""` 时按字符拆分
5. 对 Unicode 字符串按 UTF-8 字符安全拆分

例如：

```scheme
(string-split "a,,b," ",")
;; => '("a" "" "b" "")

(string-split "path::to::file" "::")
;; => '("path" "to" "file")

(string-split "中文" "")
;; => '("中" "文")
```

### Why
项目里原本已经有 `string-tokenize`，但它的语义更像“按单字符做标记化”：

1. 连续分隔符会被压缩
2. 主要面向字符分隔
3. 不适合作为日常编码中 `split` 的直觉替代

在 AI Coding 或常规开发场景下，看到：

```scheme
(string-split str sep)
```

通常会自然理解为：

- `sep` 可以是字符串
- 连续分隔符要保留空字段
- 前导和尾随分隔符也应该反映在结果里

因此这里补了一个更符合直觉的 `string-split`，而不是继续让使用者把 `string-tokenize` 当成 `split` 使用。

### How
实现方式如下：

1. 在 `(liii string)` 中导出 `string-split`
2. `str` 要求为字符串
3. `sep` 允许是字符串或字符，字符会先转成单字符串
4. 当 `sep` 非空时，使用 `string-position` 逐段查找分隔符
5. 每次命中后保留命中前的片段，并继续向后搜索
6. 不压缩连续分隔符，因此会自然保留空字段
7. 当 `sep = ""` 时，使用 `utf8-string-length` 和 `u8-substring` 按字符拆分，避免把中文拆坏

### 测试覆盖
本次新增测试覆盖以下场景：

1. 单字符分隔符
2. 多字符字符串分隔符
3. 连续分隔符
4. 前导和尾随分隔符
5. 未命中分隔符
6. 空分隔符按字符拆分
7. Unicode 内容与 Unicode 分隔符
8. 字符分隔符兼容
9. 类型错误与参数个数错误

## 2026-03-27 新增 `(liii string)` 的 `string-replace`

### What
在 `(liii string)` 中新增顶层函数：

```scheme
(string-replace str old new)
```

语义定义为：

1. 默认替换 `str` 中所有 `old` 的非重叠匹配
2. 按原字符串从左到右扫描
3. 不会重复扫描刚插入的 `new`
4. `old` 为空字符串时，返回 `str` 的副本
5. 没有匹配时，返回原内容的副本

例如：

```scheme
(string-replace "hello world hello" "hello" "hi")
;; => "hi world hi"

(string-replace "aaa" "a" "aa")
;; => "aaaaaa"
```

### Why
`srfi-13` 本身也有一个 `string-replace`，但它的语义更接近“按区间拼接子串”，并不符合大多数开发者和 AI Coding 场景下对 `replace` 的直觉。

在日常编码里，看到：

```scheme
(string-replace str old new)
```

通常会自然理解为：

- 查找 `old`
- 把所有匹配都替换成 `new`

如果继续直接暴露 `srfi-13` 的原始语义，AI 或使用者很容易写出看起来正确、实际行为却完全不同的代码。

### How
实现方式如下：

1. 在 `(liii string)` 的 import 中显式排除 `srfi-13` 的 `string-replace`
2. 导出 Goldfish 自己的 `string-replace`
3. 使用 `typed-lambda` 保证三个参数都必须是字符串
4. 使用 `string-position` 在原字符串中逐段查找 `old`
5. 通过累积片段后一次性 `string-append`，构造最终结果
6. 对 `old = ""` 特判，直接返回 `string-copy str`，避免无穷替换

### 测试覆盖
本次新增测试覆盖以下场景：

1. 基本的全部替换
2. 未命中时返回原内容
3. `new = ""` 的删除语义
4. `old = ""` 的边界行为
5. 非重叠替换
6. `new` 中再次包含 `old` 时不重复扫描
7. Unicode 字符串替换
8. 类型错误与参数个数错误

## 2026-03-29 补充 string-split 与 Python 兼容的测试用例

### What
为 `string-split` 新增与 Python `str.split()` 行为一致的测试用例，确保 Goldfish 的实现细节与 Python 保持统一。

Python 测试文件: `tests/python_split_test.py`

新增测试覆盖以下场景:

1. **单字符字符串** - `"a".split(",")` => `["a"]`
2. **多字符分隔符边界情况** - `"abc".split("bc")` => `["a", ""]`
3. **更多连续分隔符场景** - `"a,,,b".split(",")` => `["a", "", "", "b"]`
4. **分隔符重复出现（重叠匹配）** - Python 不会重叠匹配，如 `"aaaa".split("aa")` => `["", "", ""]`
5. **更多特殊字符场景** - 制表符、换行符等
6. **路径/URL 场景** - `/usr/local/bin`.split("/") => `["", "usr", "local", "bin"]`

### Why
Python 的 `str.split()` 是业界最广泛使用的字符串分割语义，确保 Goldfish 的 `string-split` 与之保持一致可以减少开发者的认知负担。

### How
1. 编写 `tests/python_split_test.py` 全面测试 Python 的 `str.split()` 行为
2. 验证所有测试用例在 Python 中的输出
3. 将对应的测试用例添加到 `tests/goldfish/liii/string-test.scm`
4. 运行 Goldfish 测试确认所有用例通过

### Python 与 Goldfish 的行为差异
- **空分隔符**: Python 的 `str.split("")` 会抛出 `ValueError`，而 Goldfish 的 `string-split` 将空分隔符视为按字符拆分（`"abc".split("")` => `["a", "b", "c"]`）。这是 Goldfish 的扩展功能，不是缺陷。
