## 2026/3/3 优化njson存储方式

### What
1. njson 状态模型重构：
   - 将原先进程级全局状态迁移为 `NjsonState`；
   - 引入 `unordered_map<s7_scheme*, NjsonState>` registry；
   - 句柄池、generation、free-list、keys 缓存全部按 VM 隔离。
2. 线程归属检查新增：
   - `NjsonState` 记录 `owner_thread_id`；
   - 新增 `njson_require_owner_thread`，在所有 `g_njson-*` 入口执行检查；
   - 跨线程访问同一 VM 的 njson API 将抛 `thread-error`。
3. keys 缓存重构：
   - 将 keys 缓存从 `s7_pointer + gc_protect/unprotect` 改为 `vector<string>`；
   - 缓存命中时再按需构建 Scheme list 返回；
   - 移除 njson keys 缓存路径中对 `s7_gc_protect/s7_gc_unprotect_at` 的依赖。
4. 调用路径与签名同步：
   - `store_njson_value`、`njson_value_by_id*` 等 helper 统一显式传入 `s7_scheme*`；
   - `glue_njson` 注册阶段显式初始化当前 VM 的 njson state。

### Why
1. 旧实现以进程级全局状态承载 njson 句柄，多个 VM 共存时存在状态串扰风险。
2. 旧实现对线程访问边界没有明确约束，在异步/并发场景下容易形成竞态与错误行为。
3. keys 缓存持有 Scheme 对象并依赖 GC protect 生命周期，复杂度高且维护成本高。
4. 需要建立清晰、可验证的并发契约：`njson` 以 VM owner-thread 为执行边界。

### How
1. 数据结构迁移：
   - 新增 `NjsonState`，收拢原 `njson_handle_store / generations / free_ids / keys_cache_*`；
   - 新增 `njson_get_or_create_state` 与 `njson_register_state` 管理 VM 级状态。
2. helper 层改造：
   - `make_njson_handle`、`extract_njson_handle_id`、`njson_value_by_id*`、`store_njson_value` 改为基于 `sc` 读取/写入对应 `NjsonState`。
3. 线程契约落地：
   - 在 `f_njson_*`、`njson_run_update`、`njson_run_merge`、`f_njson_schema_report` 增加 owner-thread 校验；
   - 违反约束时统一返回 `thread-error`。
4. keys 缓存实现替换：
   - 缓存构建：对象键收集为 `vector<string>`；
   - 缓存读取：命中后按键列表构建 Scheme list；
   - 缓存失效：沿用 `set!/drop!/merge!/deep-merge!/free` 的 invalidation 路径。

### 兼容性说明
1. API 名称与返回值语义保持不变（`njson-ref/set/append/drop/merge/schema-report` 等对外接口不变）。
2. 新增并发约束：同一 VM 上，非 owner-thread 调用 njson API 会抛 `thread-error`。
3. `njson-keys` 的缓存命中路径从“直接返回缓存的 Scheme list”改为“从 `vector<string>` 重建 list”，高频场景下会增加一定开销。
4. keys 缓存不再绑定 GC protect 生命周期，内存与对象生命周期管理更简单、更可控。
